# Provenance Guard — Planning Document

A backend system for classifying whether text-based creative content was written by a human or generated by AI. Built for creative sharing platforms to protect attribution, build trust, and give audiences the context they need.

---

## 1. Detection Signals

### Signal 1: LLM-Based Classification (Groq)
**What it measures:** Semantic and stylistic coherence assessed holistically. The Groq model evaluates whether the text reads as human-written or AI-generated based on overall narrative flow, voice consistency, naturalness of transitions, and presence of human idiosyncrasies (digressions, emotional texture, cultural references, imperfect logic).

**Output format:** A calibrated score between 0 and 1, where:
- `0` = strong AI indicators (overly polished, generic transitions, templated structure, absence of personal voice)
- `1` = strong human indicators (organic voice, natural digressions, contextual specificity, emotional texture)

**How the score is produced:** The Groq API is prompted with a structured request asking the model to assess the text on a 0-100 scale across 4 dimensions: voice authenticity, transition naturalness, contextual specificity, and structural predictability. The dimension scores are averaged and normalized to [0,1].

**Why this property differs between human and AI writing:** LLMs optimize for broad acceptability — they avoid idiosyncrasy, emotional rawness, and cultural specificity that might alienate some readers. Human writers write from lived experience, with voice shaped by personal history, region, education, and mood. This creates a detectable difference in *texture* that statistical methods alone miss.

**Blind spot / what it misses:**
- Highly edited, polished human prose (e.g., a New Yorker essay revised by 3 editors) may read as "too smooth" and score as AI - like.

- AI text heavily prompted with persona constraints ("write like a depressed teenager from Queens") can spoof voice authenticity.

- Very short texts (&lt;100 words) don't give the model enough material to assess voice consistency.

- The signal depends on Groq API availability and latency -- if the API is down or slow, the pipeline degrades.

### Signal 2: Stylometric Heuristics (Pure Python)

**What it measures:** Statistical structural properties computable without external APIs. Specifically:

- **Sentence length variance:** Standard deviation of sentence lengths (in words). Human writing has higher variance; AI tends toward uniform sentence length.

- **Type - token ratio (TTR):** Unique words / total words. Human writing shows more lexical diversity per topic; AI reuses common phrasing and transitions.

- **Punctuation density:** Punctuation marks per 100 words. Humans use punctuation idiosyncratically (em - dashes, semicolons, inconsistent comma usage); AI uses more standardized punctuation patterns.

- **Average sentence complexity:** Approximated by average dependency depth or clause count per sentence. Humans vary complexity within a paragraph; AI maintains more consistent complexity.

**Output format:** A composite score between 0 and 1, where:
- `0` = highly uniform, machine-like statistical profile
- `1` = highly variable, human-like statistical profile

**How the score is produced:** Each metric is computed from the raw text, normalized against empirically derived baselines (human text averages vs. AI text averages), then combined via weighted average:

- Sentence length variance: 30%
- TTR: 35%
- Punctuation density deviation: 15%
- Sentence complexity variance: 20%

**Why this property differs between human and AI writing:** LLMs are trained on massive corpora where the most probable next token is selected. This creates a statistical smoothing effect -- sentence lengths cluster around the mean, vocabulary stays within high-frequency bands, and punctuation follows predictable patterns. Humans are statistically "noisier" -- we write short punchy sentences next to long rambling ones, reuse favorite words while avoiding others arbitrarily, and punctuate by ear.

**Blind spot / what it misses:**
- Technical/legal writing where uniformity is *required* by genre convention may score as AI - like despite being human - written.

- Deliberately stylized AI prompts (e.g., "write with wildly varying sentence lengths") can spoof variance metrics.

- Non - native English speakers may have lower TTR and more uniform sentence structures due to limited vocabulary, triggering false positives.

- Poetry and experimental prose often violate normal statistical patterns intentionally.

**Signal availability handling:**

The system operates in two modes depending on Groq API availability:

- **Full mode (both signals available):** Groq LLM score + Stylometric score combined

- **Fallback mode (Groq unavailable):** Stylometric score only, with adjusted thresholds

**Full mode combination (both signals):**

raw_combined = (w1 * s1) + (w2 * s2)
where w1 = 0.55 (Groq LLM), w2 = 0.45 (stylometric)

**Calibration to confidence score:**

The weights favor the LLM signal slightly because it captures semantic nuance that pure statistics miss. However, the stylometric signal provides a critical independent check — if the LLM is fooled by a sophisticated adversarial prompt, the stylometric signal may still flag uniformity.

**Fallback mode (Groq unavailable):**

If the Groq API call fails (timeout, rate limit, error), the system falls back to stylometrics alone:

raw_combined = s2  (stylometric score only)

In fallback mode, the confidence thresholds are tightened to compensate for reduced signal diversity:

- Uncertain band widens: 0.30 – 0.70 (instead of 0.35 – 0.65)

- High - confidence human requires score ≥ 0.70 (instead of ≥ 0.65)

- The label text appends a disclaimer: "Assessment based on structural analysis only; semantic review unavailable."

**Asymmetry penalty (false-positive protection):**

Before calibration, apply a penalty that reflects the principle: **misclassifying human work as AI is worse than the reverse.**

if raw_combined < 0.42:
raw_penalized = raw_combined - 0.06
else:
raw_penalized = raw_combined

This penalty only applies in full mode. In fallback mode, the widened uncertain band already compensates for the lack of semantic signal -- no additional penalty is applied.

**Calibration to confidence score:**

confidence = 1 / (1 + e^(-k * (raw_penalized - midpoint)))

Parameters:

- `midpoint = 0.5` (design decision: 0.5 means "genuinely uncertain, no lean either way")
- `k = 6` (steepness -- creates meaningful separation between 0.51 and 0.95)

**Why this calibration works:** A sigmoid with k = 6 creates a soft S - curve where scores near the midpoint (0.4 – 0.6) spread out, while extreme scores (0.1 or 0.9) compress slightly. This means a 0.51 (barely above uncertain) lands in the uncertain zone, while a 0.95 (deep in human territory) lands firmly in high - confidence human. The 0.35/0.65 hard thresholds create a wide "uncertain" band that prevents the system from pretending certainty.

**Score meaningfulness verification:**
- Test with 20 known - AI samples and 20 known - human samples in both modes.

- In full mode: AI samples should cluster below 0.35, human samples above 0.65.

- In fallback mode: samples should cluster more toward the center (0.30 – 0.70) -- wider uncertainty is expected and correct.

- If 0.51 and 0.95 produce the same label category, the sigmoid steepness or thresholds need adjustment.

## 2. Uncertainty Representation

### What does a confidence score represent?

- **0.0–0.35:** High confidence AI-generated. Both signals strongly agree the text lacks human variance and organic structure.

- **0.35 – 0.65:** **UNCERTAIN.** The signals disagree, or both are near the midpoint. This is the "genuinely don't know" zone. A 0.51 and a 0.95 must produce meaningfully different labels -- 0.51 lands here, 0.95 does not.

- **0.65 – 1.0:** High confidence, written by a human. Both signals agree the text shows human-like variance and organic lexical patterns.

### Thresholds

| Score Range | Label Category | System Action |
|-------------|----------------|---------------|
| 0.00 – 0.35 | Likely AI | Apply "AI-Generated" transparency label |
| 0.35 – 0.65 | Uncertain | Apply "Uncertain" transparency label + flag for optional human review |
| 0.65 – 1.00 | Likely Human | Apply "Human-Written" transparency label |

### Why 0.5 means "genuinely uncertain"

A score of 0.5 is the calibrated midpoint where the sigmoid outputs exactly 0.5 when raw_combined equals the midpoint. It represents the point where the system has no statistically meaningful basis to lean either direction. A 0.51 is barely above this -- it stays in the uncertain zone. A 0.95 is deep in the high - confidence human zone. The 0.35/0.65 hard boundaries create a wide "uncertain" band to prevent the system from pretending certainty when it shouldn't.

### Testing score meaningfulness

Verification plan: Test with 20 known-AI samples and 20 known-human samples. Compute score distributions. If AI samples cluster below 0.35 and human samples above 0.65 with minimal overlap in the 0.35–0.65 band, the scoring is meaningful. If scores are uniformly distributed, the signals are not discriminating.

## 3. Transparency Label Design

### High-Confidence AI (score 0.00–0.35)

**Label text:**
> "This content was likely generated with AI assistance. The system detected consistent statistical patterns typical of language model output, including uniform sentence structure and repeated phrasing templates."

### High-Confidence Human (score 0.65–1.00)

**Label text:**
> "This content was likely written by a human author. The system detected natural variation in writing style, organic vocabulary choices, and thematic development consistent with human composition."

### Uncertain (score 0.35–0.65)

**Label text:**
> "We cannot confidently determine whether this content was written by a human or generated with AI. The text shows mixed signals — some patterns suggest human authorship, others suggest AI assistance. The creator may submit an appeal if they believe this assessment is incorrect."

### Design rationale

- Plain language, no jargon (perplexity, burstiness, n-grams are not mentioned).

- Each label explains *what the system detected* in human terms, not *how the system works in its entirety*.

- The uncertain label explicitly offers the appeals path -- reduces creator frustration by giving them immediate agency.

- All three variants are written as full sentences a non - technical reader on a creative platform would understand without clicking "learn more."

## 4. Appeals Workflow

### Who can submit an appeal?

Any creator whose content has been submitted through the system and received a classification (AI, Human, or Uncertain). The appeal is tied to the original submission ID.

### What information does the creator provide?

- **Original submission ID** (auto-populated from the content they are appealing)

- **Creator statement:** Free - text field (max 500 chars) explaination of why creator believes the classification is wrong. 

For example: "I wrote this poem at 3am after a breakup. The repetition is intentional, it is not derived from AI templates."

- **Optional evidence:** Links to drafts, revision history, or external documentation (not verified by system, stored for human reviewer context).

### What happens when an appeal is received?

1. **Status update:** Original content status changes from `classified` → `under_review`.

2. **Audit log entry:** Appeal is logged alongside original decision (timestamp, original score, signals, creator statement).

3. **Response to creator:** Immediate JSON response with `appeal_id`, `status: "under_review"`, and `estimated_review_time: "5-7 business days"`.

4. **Human reviewer queue:** Appeal enters a queue with original submission data, scores, signal breakdowns, and creator statement visible.

### What does a human reviewer see?

A dashboard entry showing:
- Original text
- Signal 1 score + explanation of what it measured
- Signal 2 score + explanation of what it measured
- Combined confidence score + threshold category
- Creator's appeal statement
- One - click actions: `Confirm Classification` or `Override to Human` / `Override to AI` / `Override to Uncertain`

### Automated re-classification?

No. This system does not implement automated re-classification. The human reviewer makes the final call. The audit log records the override and the reviewer's justification. This keeps the system honest about its own limitations.

## 5. Anticipated Edge Cases

### Edge Case 1: Deliberately Minimalist Human Poetry

**Scenario:** A human poet submits a 40 - word poem with heavy anaphora, simple vocabulary, and short lines. E.g., "I am here. I am there. I am nowhere. I am everywhere."

**Why the system handles it poorly:** Both signals may fire false positives. Perplexity/burstiness sees low variance (short, repetitive lines). Semantic fingerprint sees high repetition (anaphora). The raw combined score may drop into the 0.20–0.30 range, triggering a high-confidence AI label.

**Mitigation in design:** The wide uncertain band (0.35 – 0.65) catches some of these, but extreme minimalism may still fall below 0.35. The appeals workflow is the primary safety net -- the creator can explain the intentional repetition. Future improvement: add a third signal for stylistic intent detection (e.g., recognizing poetic devices).

### Edge Case 2: AI Text Heavily Edited by a Human

**Scenario:** A creator uses AI to generate a first draft, then heavily edits it -- changing 60% of sentences, adding personal anecdotes, restructuring paragraphs.

**Why the system handles it poorly:** The text is genuinely hybrid. Signal 1 may detect human burstiness in the edited sections, but Signal 2 may still find AI template residue in the unedited 40%. The scores may conflict, landing in the uncertain zone (0.40–0.60). This is actually correct behavior -- the system *should* be uncertain about hybrid content -- but creators may find "uncertain" stigmatizing when they did substantial human work.

**Mitigation in design:** The uncertain label explicitly says "mixed signals" rather than "probably AI." The appeals workflow allows creators to argue for human override. Future improvement: detect edit boundaries (requires diff against known AI outputs, which is not in scope).

### Edge Case 3: Adversarial Flooding Attack

**Scenario:** A malicious actor submits thousands of AI - generated pieces with adversarial perturbations designed to spoof human signals (e.g., injecting random typos, forcing sentence length variance).

**Why the system handles it poorly:** Rate limiting prevents volume, but sophisticated adversarial text may bypass both signals if the perturbations are carefully calibrated. The system has no adversarial training data.

**Mitigation in design:** Rate limiting (see below) caps submission volume per IP/API key. The audit log enables post - hoc forensic analysis. The system does not claim perfection -- the README and labels explicitly state this is a best - effort detection system.

### Flow Narratives

**Submission Flow (1):**
A piece of text arrives at `POST /submit`. The rate limiter checks if the client (by IP or API key) is within allowed limits. If yes, the text enters the detection pipeline. Signal 1 analyzes perplexity and burstiness, outputting a raw score. Signal 2 analyzes lexical diversity and semantic coherence, outputting its raw score. The confidence scorer combines these via weighted harmonic mean with asymmetry penalty, then calibrates through a sigmoid. The label generator maps the confidence score to a category (AI / Uncertain / Human) and selects the corresponding verbatim text. The entire decision — all scores, the label, and metadata — is written to the audit log. Finally, the API returns a JSON response with the attribution result, confidence score, and label text to the platform.

**Appeal Flow (2):**
A creator submits `POST /appeal` with the original submission ID and their statement. The system validates the submission ID exists and is not already under review. The original content's status updates to `under_review`. An appeal record is created with the creator's statement. Both the status change and the new appeal are appended to the audit log. The system returns an appeal ID and confirmation to the creator. The appeal enters the human reviewer queue with all original decision data visible. The reviewer makes a final determination, which is logged, and the content status updates to `resolved`.

## Rate Limiting Configuration

**Endpoint:** `POST /submit`

**Limits chosen:**
- **100 requests per 15 minutes per IP address**
- **500 requests per hour per API key** (for authenticated platform integrations)

**Reasoning:**
A typical human creator on a writing platform might submit 1–3 pieces per day for analysis. 100 requests per 15 minutes allows legitimate bulk operations (e.g., a platform migrating a back catalog) while preventing adversarial flooding. The API key limit is higher for platform integrations that serve many users. An adversary trying to flood the system with AI-generated content to poison or overwhelm it would hit the IP limit quickly. The 15-minute window is short enough to not frustrate legitimate users but long enough to require an adversary to distribute across many IPs to be effective.
